# ai-gm-core 日本語まとめ（レイナ＆シュン用）

## これは何か

ai-gm-core は「ゲーム」を作るためのプロジェクトではない。

**AIを“ゲームマスター（GM）”として扱うための、思考の芯（コア）を置く場所**である。

目的は、

* 勝ち負けを決めること
* 正解を提示すること
  ではなく、

**プレイヤーの行動を解釈し、
世界がどう変わったかを“物語として返す”こと。**

---

## 一番重要な考え方

### 行動そのものがプロンプトになる

プレイヤーはAIに命令文を書く必要はない。

* 何をしたか
* 何をしなかったか
* どこで迷ったか
* 何を避け続けたか

**これらの行動ログ自体が、AIへの入力（プロンプト）になる。**

AIはそれを「正しい／間違い」で評価しない。

**その人の意図・性格・立場として解釈する。**

---

## 世界は「放っておいても動く」

プレイヤーが何もしなくても、

* 時間は進み
* 状況は変化し
* 他者（または世界）は判断を下す

「不介入」「保留」「沈黙」も
**立派な選択として世界に影響する。**

---

## 同期はしないが、共有はする

* リアルタイム同期はしない
* MMOのような完全共有も目指さない

代わりに、

**他人の行動や判断を「エピソード」として受け取る。**

* 誰かが介入しなかった結果
* 別の判断をした人物の話
* 世界史の一節のような出来事

これにより、
「同じ瞬間」ではなく
**「同じ世界の痕跡」を共有する。**

---

## なぜスタンドアロンで成立するのか

この仕組みは、

* グラフィック
* 数値バランス
* 高速同期

を必要としない。

必要なのは、

* 行動ログ（テキスト）
* AIによる解釈
* 物語としての返答

つまり、
**最小構成でも“体験”が成立する。**

---

## 現在地メモ（2025-12-24）

※ 本セクションは、ai-gm-core の設計議論がどこまで進んだかを
　レイナとシュンさんが思い出すための「途中経過ログ」です。

---

### いま何を決めたか

* **ai-gm-core は Node.js ベースで進める**

  * 有志参加・OSS展開を考えると心理的ハードルが低い
  * 実装より「試せる」「触れる」を優先

* **HTTP サーバーは Fastify を採用する方向**

  * Express でも成立はするが、

    * 軽量
    * スキーマ前提
    * 将来の拡張（AI I/O・イベント化）
      を考えると Fastify の思想が ai-gm-core と噛み合っている

* **これは“ゲームサーバー”ではない**

  * リアルタイム同期を目的としない
  * セッション管理や高速マッチングは主眼ではない
  * 役割は「行動ログを受け取り、世界の変化を返す GM」

---

### 想定している最小構成（スタブ）

> 「動いてるかどうか」が分かれば十分

1. クライアント（仮）

   * ゲームエンジンでも
   * CLI でも
   * テキスト送信でもよい

2. ai-gm-core（Node + Fastify）

   * 行動ログを受け取る
   * 文脈（過去ログ・世界状態）をまとめる
   * AI に渡す

3. AI（OpenAI API / Ollama 等）

   * 判断を下す
   * 世界がどう変わったかを「物語」として返す

4. レスポンス

   * 数値ではなくテキスト
   * 成功／失敗ではなく「結果の描写」

---

### 重要な前提（再確認）

* **行動そのものがプロンプト**

  * 命令文を書かせない
  * プレイヤーの沈黙・放置・回避も意味を持つ

* **世界は勝手に進む**

  * プレイヤー入力がなくてもイベントは発生する
  * GM（AI）は「待たない」

* **完全同期はしない**

  * MMO を目指さない
  * 他人の行動は「同時体験」ではなく「エピソード」として共有

---

### Unreal / PS5 との距離感

* 最終的に Unreal Engine から呼ばれる未来は否定しない
* ただし **最初からエンジン統合はしない**
* ai-gm-core は

  * Unreal / Unity / Web / CLI
  * どこからでも叩ける「GMの頭脳」であるべき
